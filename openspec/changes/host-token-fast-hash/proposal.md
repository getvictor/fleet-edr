# Verify host tokens with a fast keyed hash, not argon2id

## Why

Every authenticated agent request (`POST /api/events`, `GET /api/commands`, `PUT /api/commands/{id}`) runs the host-token verification in `server/endpoint/internal/mysql/hash.go`, which recomputes argon2id (`t=3, m=64 MiB, p=4`) fresh on every call. There is no verification cache: `Store.VerifyWithMeta` calls `verifyToken` (a full `argon2.IDKey`) on every request.

Production telemetry on prod-render (SigNoz, 6h window) shows the cost directly. Both authenticated agent endpoints sit at a p50 of 235ms (`POST /api/events`) and 251ms (`GET /api/commands`), while every DB span in those requests is sub-millisecond to single-digit-millisecond (`sql.stmt.exec` p50 0.5ms, `sql.tx.commit` p50 4.6ms) and the unauthenticated `/readyz` is 0.5ms. The ~235ms that is not in any DB span is the argon2id computation: in a trace it appears as an untraced gap between the `enrollments` auth SELECT finishing and the ingest transaction beginning. At a 235ms baseline, any request that hits CPU or memory-bandwidth contention from concurrent agent polling on Render's shared CPU crosses the 500ms slow-log threshold, which is the steady drip of `http request (slow)` WARN logs that prompted this change (395 `/api/events` + 65 `/api/commands` slow logs in 6h).

The choice of a memory-hard password KDF is wrong for this input. argon2id, bcrypt, and scrypt exist to protect low-entropy, human-chosen passwords against offline brute force. The host token is a 32-byte random secret (`generateToken` reads `crypto/rand`), not a password. A high-entropy random token is not brute-forceable, so the slow KDF buys nothing and costs ~235ms per request. The industry-standard storage for random API bearer tokens (GitHub, Stripe) is a single fast hash. The code even already computes `tokenID = SHA-256(token)` as the indexed lookup key, so the token is unambiguously a random secret.

The in-code comment claims steady-state auth is "a constant-time compare against the stored hash, not a fresh hash, so steady-state auth is microseconds." The production data contradicts it: there is no cache and steady-state is 235ms. So this is also a fix against the documented intent, not only a tuning change.

## What changes

- **Host-token verification switches from argon2id to keyed HMAC-SHA256.** `hashToken` / `verifyToken` compute `HMAC-SHA256(pepper, token)` and compare constant-time. Verification drops from ~235ms to sub-microsecond, removing the latency floor on every authenticated agent request. The keyed construction (a server-held pepper) means a read-only database leak yields neither the token (the stored value is a one-way hash of a high-entropy secret) nor the ability to forge one (the attacker lacks the pepper).
- **The per-row salt is removed.** HMAC keys on a single server-wide pepper, not a per-row salt. The `host_token_salt` and `previous_host_token_salt` columns are dropped. `tokenID` (the SHA-256 indexed lookup key) and the previous-token grace window are unchanged.
- **Keys are derived from a single server root key via HKDF, not provisioned per purpose.** A new required `EDR_SECRET_KEY` (high-entropy, sourced like the other server secrets with the `*_FILE` Docker-secret fallback, added to `render.yaml` with `generateValue: true`) is the one root secret. A small internal `keyring` package derives per-purpose keys from it with HKDF-SHA256 (`crypto/hkdf`, stdlib in Go 1.26) under a versioned domain-separation label. This is the standard key-separation pattern (Rails `secret_key_base`, Django `SECRET_KEY`): future features that need a server-side key call `Derive("edr/<purpose>/v1")` and add no new boot-time env var, while HKDF's PRF property keeps every derived key cryptographically independent of the others and of the root. Per-record random values (nonces, one-time tokens, the per-token salt being removed here) stay `crypto/rand` generated; only long-lived server keys are derived.
- **`EDR_SESSION_SIGNING_KEY` is folded into the root and removed.** The pre-auth cookie signing key (OIDC state cookie + break-glass WebAuthn challenge cookie) is now `keyring.Derive("edr/session/signing/v1")` rather than its own env var; the host-token pepper is `keyring.Derive("edr/host-token/pepper/v1")`. `EDR_SECRET_KEY` is required on every boot (the pepper is always needed), replacing the previous "session key required only when OIDC/break-glass configured" gate. This is a breaking config change: deployments must set `EDR_SECRET_KEY` and drop `EDR_SESSION_SIGNING_KEY`.
- **This is a breaking change with no token re-hash migration.** Existing enrollment rows hold argon2id hashes that will never match an HMAC, so every currently enrolled host gets a 401 on its next request and re-enrolls automatically via the existing "Re-enrollment on token revocation" path. This is an accepted one-time fleet-wide re-enroll to keep the change simple; no data migration is performed.
- **The argon2id test-cost workaround is deleted.** The package-level `argonTime` / `argonMemory` / `argonThreads` vars and the `init()` that lowers them under `go test` (the issue #170 workaround) go away, since HMAC is fast in every build. This is a net simplification of the package.

### Not in this change

- **Operator password storage stays on argon2id.** The `ui-authentication-session` "Passwords are stored as argon2id hashes" requirement is correct: operator passwords are low-entropy and a memory-hard KDF is the right tool there. This change touches only the high-entropy host bearer token.
- **A token re-hash / dual-read migration.** Explicitly out of scope by decision: the breaking re-enroll is accepted.
- **A verification cache.** Made unnecessary by the fast hash; not added.
- **Any change to token issuance entropy, lifetime, rotation, or the grace window.** `generateToken`, `EDR_HOST_TOKEN_LIFETIME`, and `EDR_HOST_TOKEN_GRACE` are unchanged.

## Resolved decisions

1. **Key sourcing: one root key plus HKDF derivation, not a dedicated per-purpose secret.** Anticipating many future server-side keys, a per-purpose env var (`EDR_HOST_TOKEN_PEPPER`, then one more per feature) proliferates required boot secrets, and raw reuse of one key for unrelated purposes couples them. The key-separation pattern (single `EDR_SECRET_KEY` root, HKDF-SHA256 with a versioned domain-separation label per purpose) gives the single-secret operator experience with cryptographically independent derived keys. New features add a label, not an env var. The existing session signing key is folded in as the pattern's second consumer and `EDR_SESSION_SIGNING_KEY` is removed.
2. **Drop the salt columns.** Confirmed: `host_token_salt` / `previous_host_token_salt` are dropped via migration. HMAC keys on the derived pepper, not a per-row salt; a `NOT NULL VARBINARY` column cannot be left behind without a migration anyway, and a stale unused column invites confusion.
