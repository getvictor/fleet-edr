## Why

Today a non-analyst operator can only reach the console through OIDC just-in-time provisioning (auto-created on first SSO sign-in, landing in `analyst` per `DefaultJITRole`) or the break-glass account. The admin Users page (#135) manages existing users only; its subtitle states the constraint: "New users arrive through single sign-on or break-glass." Onboarding a senior analyst, auditor, or admin is therefore a two-step manual dance: the person signs in via SSO (landing in `analyst`), then an admin promotes them through the Users page or by hand-written SQL.

There is no way for an admin to pre-provision a user: stage an email plus role now, before that person has ever signed in, so they land in the correct role on their first SSO login. This was deliberately deferred in wave 1 and wave 2: the `user.invite` action constant already exists and is granted to `admin`, `bootstrap_tokens.user_id` was made nullable for a future invite token, and the archived change specs record "manage existing users only (no invite flow)". This change builds the manual, admin-driven complement to OIDC group mapping (#136): the EDR is the source of truth for the pre-assigned role.

## What Changes

- Add an audited create endpoint `POST /api/settings/users` under the existing operator-session plus CSRF boundary, gated on the existing `user.invite` action (already registered and granted to `admin`). Body `{"email": "...", "role": "<bindable role>"}`. It transactionally inserts a `users` row in a new `provisioned` lifecycle state with one global `role_bindings` row for the chosen role and no usable credential. It rejects `super_admin` for a non-super-admin actor (mirroring the role-edit and service-account rules) and rejects a duplicate email.
- Model the pre-provisioned account as an explicit lifecycle state: add `provisioned` to the `users.status` ENUM, alongside `active` and `disabled`. This mirrors the IAM-industry lifecycle model (Okta `STAGED`, Entra, AWS IAM Identity Center) where "staged but never authenticated" is a first-class, queryable, auditable status rather than a property inferred from the absence of an identity row (which would falsely flip an active user back to "pending" whenever an OIDC identity is rebound or rotated).
- Reconcile on first login: when a pre-provisioned email next authenticates via OIDC, the JIT provisioner adopts the existing `users` row instead of creating a fresh one or returning an email conflict. It links the new OIDC identity to that row, flips the status `provisioned` to `active`, and keeps the pre-assigned role rather than binding `DefaultJITRole`. Matching is on the verified email claim only. Adoption works regardless of the `allow_jit` toggle: pre-provisioning is an explicit admin staging action, not just-in-time creation, so binding an identity to an already-staged, admin-approved account is always honored; `allow_jit` continues to govern only the auto-creation of brand-new unknown subjects.
- Audit every pre-provisioning with a dedicated `user.provisioned` action recording the acting operator, the target user, and the assigned role. A dedicated verb (rather than overloading `authz.role_binding.create` with a payload marker) keeps SIEM correlation and alerting keyed on a stable, semantically precise event type, distinct from OIDC JIT (`user.created`, source `oidc.jit`) and admin role edits (`authz.role_binding.*`).
- Add an "Add user" control to the Users page (`Users.tsx`) opening an email plus role form, gated on `user.invite` via the `useCan()` seam and wired through a new `api.ts` helper that refreshes the list. A pre-provisioned row (status `provisioned`) renders a distinct "Pending" badge so it is visually distinguishable from active users.

Out of scope (deferred): OIDC group-claim to role mapping (#136); email delivery and invitation links (a token-redemption invite flow leveraging the nullable `bootstrap_tokens.user_id` is a later wave); local-password account creation (no local-credential path exists); multi-tenant or host-group scope assignment (#137); a delete or revoke-invite endpoint for a staged user (the existing role-edit surface already operates on a provisioned row).

## Capabilities

### Modified Capabilities

- `server-identity-authorization`: add the audited `POST /api/settings/users` create endpoint gated on `user.invite`, the `provisioned` user lifecycle state, the `user.provisioned` audit action, and the first-login reconciliation that adopts a pre-provisioned account into its pre-assigned role.
- `web-ui`: add the "Add user" control to the user-management page, gated on `user.invite`, and the pending-state presentation.

## Impact

- Code: `server/identity/migrations/00007_user_provisioned_status.sql` (additive ENUM value), `server/identity/api/audit.go` (+`user.provisioned`), `server/identity/api/types.go` (+`ErrEmailExists`), `server/identity/internal/users/users.go` (+`CreateProvisioned`, +`Activate`, +status constants; the create path inserts the `principals` spine row per ADR-0017), `server/identity/internal/rbac/role_bindings.go` (+`ProvisionUser`), `server/identity/internal/oidc/jit.go` (first-login reconciliation, gated on the `provisioned` status), `server/identity/internal/useradmin/handler.go` (+create handler), and the React UI (`ui/src/permissions-core.ts`, `ui/src/api.ts`, `ui/src/components/Users/Users.tsx` + `.scss`).
- Data: one additive migration that widens the `users.status` ENUM to include `provisioned`. No data backfill; existing rows keep `active` or `disabled`. Rollback leaves the wider ENUM harmlessly in place.
- APIs: one new authed route, `POST /api/settings/users`. The session-protected allowlist is auto-derived (#463), so the route is covered without a `main.go` edit. No change to the agent protocol, the event schema, or any existing endpoint.
- Audit invariant: every applied pre-provisioning emits one `user.provisioned` row recording the acting operator, the target user, and the role. First-login adoption is covered by the standard `auth.login.success` row plus the original `user.provisioned` row.
- Rollback is a code revert; the widened ENUM is left in place. Deployments keep working with the existing OIDC JIT and break-glass onboarding paths.
