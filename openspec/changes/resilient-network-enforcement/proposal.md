# Make the inline DNS/network path a resilient enforcement point

## Why

On 2026-06-20 an endpoint lost all DNS resolution while raw-IP connectivity stayed up (`ping 8.8.8.8` worked, every name lookup failed). The network extension's `DNSProxyProvider` logged 974 `Upstream UDP connection failed` errors clustered in one minute, and the only fix was a reboot. Root cause: the DNS proxy claims every DNS flow (`handleNewFlow` returns `true`), which makes it the sole resolver for the whole machine, and it has no forward deadline and no health-driven recovery. When its upstream forwarding wedged, every claimed query went unanswered, so DNS died with no self-heal. A controlled repro on edr-dev confirmed the mechanism and one more thing: when a working upstream returns, the proxy resolves again immediately, so the incident was a genuine internal wedge that never recovered on its own, not a transient network blip.

The naive fix is "make the DNS proxy a passive observer so it can never break DNS." That is the wrong call for this product. The EDR's reason to exist (versus an osquery-style read-only agent) is detection AND response: isolating a compromised host and blocking malicious destinations are core capabilities, and they require being inline on the flow path. The codebase already enforces inline at the process layer: `extension-application-control` denies a process exec via an Endpoint Security AUTH_EXEC verdict on an explicit BLOCK match, drives that from a persisted policy snapshot that is restored on restart, and keeps every decision deadline-guarded and non-blocking. The network path should be held to that same standard, not demoted to monitoring.

So the fix is not to leave the path; it is to give the inline path correct, policy-aware failure semantics and a watchdog, mirroring how `extension-application-control` already behaves. The governing principle, made explicit:

- **Observation must never gate availability.** Telemetry capture and upstream-forwarding health are independent from the user's connectivity. A capture or forward failure on a flow that no policy blocks MUST fail open (resolve normally). This mirrors application-control's "a cold leaf-cert lookup yields an absent value without blocking".
- **Enforcement is an explicit, positive decision, and it is cheaper than forwarding.** A flow is denied only on an explicit match. A DNS block needs no upstream and no network: the proxy synthesizes the denial answer locally. So a degraded proxy that cannot reach upstream can STILL enforce blocks; what it loses is the ability to forward allowed traffic. The security function survives even when the network function degrades.
- **Bounded everything.** Every upstream forward carries a deadline and a defined outcome on timeout. The proxy never pins a flow indefinitely waiting on an upstream that will not answer.

## What changes

- **Forwarding resilience (the immediate incident fix).** Every upstream forward in the DNS proxy gets a bounded deadline (UDP has none today; only the TCP path has a 30s linger). On forward failure or timeout for a query no active policy blocks, the proxy closes the flow cleanly and fails open rather than leaving the client hung; telemetry remains best-effort and never blocks resolution.
- **Health watchdog with safe bypass (self-heal, no reboot).** The extension tracks upstream-forwarding health over a window. On sustained failure with no active network-enforcement policy, it bypasses DNS proxying so resolution returns to the system resolver, then periodically retries restoring the proxy. With an active enforcement policy it does NOT open-bypass (that would let blocked domains resolve); it rebuilds the proxy and keeps blocked domains denied during recovery. This is the guarantee that a monitoring-path wedge can never again require a reboot.
- **A network-response enforcement model.** A new capability `extension-network-response` states the on-device decision and failure contract that domain blocking and host containment plug into: default forward, deny only on explicit match, blocks answered locally without upstream, and host network containment expressed as a declarative content-filter ruleset that survives a provider restart and preserves a management lifeline to the EDR server (the same shape as CrowdStrike Falcon "Network Contain", and the same persist-and-restore shape application-control already uses).
- **The disable lever fails fast.** The incident showed `disable-dns-proxy` hangs from a non-GUI SSH session because `NEDNSProxyManager.loadFromPreferences` waits on a `nesessionmanager` round-trip that never returns. The break-glass recovery must not itself hang during an outage: bound the preferences operation and report an actionable failure instead of blocking forever. The runtime watchdog is the primary automatic recovery; this keeps the manual lever usable as a backstop.

### Not in this change

- **Network-response policy distribution.** The server API, snapshot wire format, and UI to author and push a domain blocklist or to contain or release a host are out of scope. This change specifies only the on-device decision and failure semantics those will target, so the policy plane can be built against a stable contract (the same way application-control's on-device decision predates and anchors its server-side policy surface).
- **Process and file response actions** (kill, quarantine a binary) beyond what `extension-application-control` already covers.
- **Encrypted-DNS (DoH/DoT) interception** beyond the existing TCP DNS path. Naming it as a known gap, not solving it here.

## Impact

- Affected capabilities: `endpoint-event-collection` (MODIFIED: DNS query capture forwarding semantics), `extension-network-response` (ADDED: new capability), `host-app-extension-manager` (ADDED: bounded disable lever).
- Affected code: `extension/edr/networkextension/DNSProxyProvider.swift` (forward deadlines, fail-open on forward failure, health accounting), a new health-watchdog seam in the network extension, `extension/edr/edr/main.swift` + `ExtensionManagerLogic.swift` (bounded `disable-dns-proxy`/`enable-dns-proxy`). The enforcement-decision and containment-ruleset surfaces are new but their policy inputs are stubbed/empty until the deferred policy plane lands, so default behavior is unchanged (forward everything) except for the resilience fixes.
- No server, schema, or wire-format change in this proposal. The DNS proxy is exercised on a live macOS VM (system/VM layer) before RC, per the agent/extension ESF/NE rule.
- Warrants a NEW ADR (next number 0014) recording the fail-open-observation / explicit-block-enforcement / declarative-containment principle as a cross-cutting decision. No existing ADR captures it: ADR-0008 is scoped to Endpoint Security subscription choices and ADR-0007 to XPC peer validation, so this is a fresh decision, not an amendment. Flagged as a follow-up, not authored here.
