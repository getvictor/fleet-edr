## Why

The detection processor is the per-replica scale bottleneck (#535). A single replica falls behind ingest at a few hundred hosts: pprof on a saturated single replica shows only ~25% CPU utilization with ~74% of on-CPU time in `syscall`/`runtime.kevent` (netpoll), i.e. the single processor goroutine is blocked on per-event database round-trips, not compute. A 500-event batch is ~1000 to 1500 sequential, single-goroutine DB round-trips because `graph.(*Builder).ProcessBatch` loops per event and issues a `GetProcessByPID` (SELECT) plus an `UpdateProcessExec`/insert (UPDATE/INSERT) for each, every statement acquiring a pooled connection with a `ResetSession`. Per ADR-0016 the per-event processing cost is independent of how events are delivered, so this work lands in the part a streaming migration keeps (the builder + rule eval); doing it now unblocks the 10-to-500-endpoint pilot envelope on one or two small replicas without a premature streaming dependency.

## What Changes

- **The graph builder materializes a batch with a small constant number of database round-trips instead of one-or-more per event.** It bulk-loads the candidate process rows for every `(host_id, pid)` referenced in the batch in one query, folds the batch against an in-memory overlay that reproduces the exact per-event resolution semantics (timestamp ordering, PID-reuse closure, exec-in-place update, re-exec chain linkage, exec-without-fork synthesis, snapshot dedup, the exit-before-snapshot-exec buffer), then flushes the result as set-based writes (a multi-row INSERT for new rows, batched `CASE`-keyed UPDATEs for modified rows) on a single pinned connection. The resulting process forest is identical to the per-event path for any batch.
- **Poison-event isolation is preserved under batched persistence.** A malformed payload (JSON syntax/type error) is still dropped during the fold before it can mutate state. A value the database can never store (a data-integrity violation, e.g. the issue #379 uid/gid overflow) is isolated at flush: a batched write that fails with a permanent data error degrades to per-row execution for the failing group, dropping the offending row(s) and applying the rest, so one poison row still cannot wedge the batch. A transient fault still fails the whole batch for retry.
- **A replica runs multiple concurrent processor workers.** Instead of a single `Processor.Run` loop, a replica runs a small fixed number of worker goroutines that each claim disjoint event batches through the existing `SELECT ... FOR UPDATE SKIP LOCKED` claim (ADR-0011), the same lock-free claim the multi-replica design already relies on. This turns the idle-CPU (25%) box from ~1 core into ~all cores. The workers share one graph `Builder` (so the cross-batch exit-before-snapshot buffer stays coherent) and one detection `Engine`.
- **The shared MySQL connection pool gains a compiled ceiling.** Intra-replica worker concurrency multiplied across replicas could otherwise exhaust MySQL `max_connections`, since the pool was previously unbounded. `OpenDB` now sets a fixed `SetMaxOpenConns`/`SetMaxIdleConns`, sized to the worker count with headroom for request-path queries.
- **Unchanged:** the `processes` wire/persistence shape, the event envelope, the detection-rule output for the efficacy corpus, the claim/ack/nack queue contract, and the leader-gating of the periodic janitors. The new worker concurrency and pool ceiling are internal operational tuning fixed as compiled constants, not new environment variables, consistent with the minimal-configuration-surface requirement.

### Modified Capabilities

- `server-process-graph-builder`: a new requirement that batch materialization is set-based and produces a forest identical to per-event application, and a clarification that poison-event isolation holds under batched persistence via a per-row flush fallback.
- `server-availability`: the "processor scales across replicas via SKIP LOCKED" requirement is broadened so the same lock-free claim also fans the processor across concurrent in-process workers within a replica, and a new requirement bounds the shared connection pool so worker concurrency cannot exhaust the database.
- `server-configuration`: the fixed-constants enumeration is extended to name the process-graph worker concurrency and the database connection-pool ceiling, keeping them off the environment surface.

## Impact

- **Affected specs:** `server-process-graph-builder` (1 added requirement), `server-availability` (1 modified, 1 added requirement), `server-configuration` (1 modified requirement).
- **Affected code:** `server/detection/internal/graph/` (builder folds over an in-memory batch session; new `batch.go`), `server/detection/internal/mysql/processes.go` (bulk-load by `(host_id, pid)` set, set-based INSERT/UPDATE flush with a per-row permanent-error fallback), `server/detection/internal/pipeline/` (processor + runner run N workers sharing one builder/engine), `server/detection/bootstrap/bootstrap.go` and `server/config/config.go` + `server/cmd/fleet-edr-server/main.go` (new `ProcessConcurrency` constant plumbed via `Deps`), `server/bootstrap/db.go` (pool ceiling).
- **Preserved invariants:** ADR-0010 (stateless server; the in-memory batch session lives within one request/claim lifetime and is discarded), ADR-0011 (multi-replica lock-free claim, now also intra-replica), and the process-graph-builder requirements (output equivalence is the acceptance gate). ADR-0016 (the processing cost is delivery-substrate-independent, so this survives a future streaming cutover).
- **Verification:** a differential property test asserts the batched fold produces byte-identical process forests to the per-event path over randomized fork/exec/exit/re-exec/snapshot batches; the efficacy corpus must show no detection regression; the #203 scale harness re-baselines a single replica at the 80/20 mix with the `--backlog-dsn` backlog gate and records how far past ~300 hosts one replica now sustains with the `event_queue` backlog bounded.
